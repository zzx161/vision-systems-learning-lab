# Linux 性能排查清单

## 适用场景

当你遇到下面这些现象时，可以从这张清单开始：

- 程序 CPU 很高
- 程序看起来卡住
- 系统偶发抖动
- 延迟突然变差
- 多线程吞吐不如预期

## 第一步：先分类型

先别着急下结论，先判断更像哪一类：

1. CPU 忙
2. 等待多
3. 内存压力
4. 系统调用卡住
5. 抖动 / tail latency 问题

## 第二步：先看全局

先问：

- 是整个系统有压力，还是只有一个进程有问题？
- 是持续问题，还是偶发问题？

建议先看：

- `top`
- `htop`
- `vmstat`

你要先得到一个粗结论：

- 整体忙不忙
- 内存紧不紧
- 是否存在明显等待

## 第三步：再盯目标进程或线程

建议看：

- `pidstat -t`
- `top -H -p <pid>`

重点看：

- 哪个线程最忙
- 哪个线程上下文切换多
- 哪个线程像关键路径线程

## 第四步：判断是在算，还是在等

如果线程高 CPU：

- 更像热点计算
- 也可能是空转

如果线程低 CPU 但系统不前进：

- 更像等待
- 可能等锁
- 可能等 IO
- 可能卡在系统调用

这时建议上：

- `strace -p <pid>`

重点看：

- `futex`
- `poll`
- `read`
- `ioctl`
- `recv`

## 第五步：怀疑性能热点时

建议看：

- `perf top`
- `perf stat`
- `perf record`

重点不是一上来就把所有指标看全，而是先回答：

- 时间花在谁身上
- 是算慢还是访存慢
- 是某个函数热，还是整段链路都慢

## 第六步：判断是平均问题还是尾部问题

如果平均值还行，但用户体验差，优先怀疑：

- 抖动
- 调度
- tail latency
- queue 短时积压

这时要特别警惕：

- 偶发 spike
- 关键线程迁移
- 锁竞争

## 最后输出什么

每次排查结束，至少留下：

1. 现象描述
2. 第一怀疑对象
3. 用了哪些工具
4. 看到的关键证据
5. 下一步最值得做什么

## 一页速记

- CPU 高：先看 `top` / `pidstat`
- 像卡住：先看 `strace`
- 怀疑整体资源压力：先看 `vmstat`
- 怀疑热点：上 `perf`
- 平均正常但体验差：重点看 tail 和抖动
