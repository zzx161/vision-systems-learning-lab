# 第 15 课：稳定性、可观测性体系与量产工程

## 为什么这节课很重要

真正拉开差距的工程师，往往不是“会修一个 bug”的人，而是：

- 能减少一类问题
- 能让问题更早暴露
- 能让团队更容易定位问题
- 能把偶发故障变成可重复分析的对象

这就是稳定性工程和量产工程的价值。

## 这节课你要学什么

学完以后，你应该能：

1. 理解稳定性不是一个单点指标，而是一整套工程能力
2. 理解可观测性为什么是系统设计的一部分
3. 区分“修现象”和“建体系”
4. 知道回归、监控、日志、健康检查该放在什么位置
5. 写出一套可重复执行的问题定位流程

## 什么叫稳定性

稳定性不是“系统今天没挂”。

更接近的理解是：

- 在长期运行中输出是否可预测
- 在边界条件下是否还能维持合理行为
- 出问题时能不能快速定位
- 改完以后会不会引入新的回归

所以稳定性同时包含：

- 运行稳定
- 性能稳定
- 结果稳定
- 排障效率稳定

## 什么是可观测性

可观测性不是“打很多 log”。

更准确地说：

当系统出了问题时，你能不能从已有信号里推断出它内部发生了什么。

这要求你提前设计好可观察信号，例如：

- 时间戳
- queue depth
- buffer 使用情况
- drop count
- 阶段耗时
- 错误码
- 关键状态转移

如果这些信号没有提前设计，很多问题出了以后就只能猜。

## 为什么量产工程很难被替代

因为它要求你同时理解：

- 硬件
- 软件
- 运行环境
- 用户场景
- 回归风险

这不是单纯代码生成能解决的。
尤其是当系统已经变大、模块已经很多时，稳定性工程的价值会越来越大。

## “修一个问题”和“减少一类问题”的区别

举个例子：

### 只修一个问题

- 某个机型偶发掉帧
- 你找到一个触发条件
- 打个 patch 修掉

### 减少一类问题

- 你顺手补上 queue depth 监控
- 补上关键阶段时间戳
- 补上 drop reason 分类
- 补上回归用例

第二种做法的价值通常远高于第一种。

## 相机系统里最应该长期监控的东西

1. 输入帧率
2. 输出帧率
3. 各阶段延迟
4. queue depth
5. drop count 与 drop reason
6. buffer 使用率
7. CPU / 内存占用
8. 多流同步误差

这些指标不是为了好看，而是为了在问题发生时有抓手。

## 健康检查为什么重要

健康检查不是“系统还活着吗”这么简单。

它更应该回答：

- 数据还在流吗
- 输出还在更新吗
- 延迟是不是已经严重偏离
- 某个模块是否进入异常但未崩溃状态

很多系统最难受的问题，不是直接 crash，而是“看似活着，其实已经不正常”。

## 回归思维

回归思维的核心是：

“我这次修复，会不会在别的条件下带来新的问题？”

你以后做系统修改时，要习惯问：

1. 哪些正常路径可能被影响？
2. 哪些边界条件可能被影响？
3. 哪些时序关系可能被改变？
4. 哪些性能指标可能被拖差？

这会让你从“功能思维”逐步转向“系统思维”。

## 为什么很多团队会长期卡在“总是在救火”

因为系统一旦缺少下面这些东西，就很容易反复掉进同一类坑：

- 关键指标没有长期记录
- 问题出现时没有分类
- 修复后没有补回归
- 排障过程没有沉淀成模板

结果就是：

- 每次都像第一次出问题
- 每次都从头猜
- 修复速度越来越依赖个别人

这也是为什么稳定性工程常常是团队能力上限的一部分。

## 一条很适合你现在岗位的稳定性思路

你现在做相机和图像链路，很适合建立这样一条思路：

1. 先让关键路径可见
2. 再让关键指标可记录
3. 再让关键异常可分类
4. 最后把排障流程写下来

为什么这条路特别适合你？

因为你已经离真实数据链路很近了。
往前再走一步，就是把“会接链路”升级成“会把链路做稳”。

## 一个值得长期保留的排障模板

当线上出现问题时，你可以按这个顺序走：

1. 明确症状是什么
2. 确定影响范围
3. 查关键时间点的指标
4. 看数据流是否中断
5. 看是否存在积压
6. 看错误集中在哪一层
7. 区分“偶发一次”还是“可复现模式”
8. 修复后补监控与回归

这类模板一旦形成，会非常有价值。

## 常见误区

### 误区 1

问题修掉了，就结束了。

### 误区 2

log 足够多，就等于可观测性够好。

### 误区 3

稳定性是测试同学的事情。

### 误区 4

偶发问题没法体系化处理。

## 你现在先记住什么

先记住这 4 句话：

1. 稳定性是一整套工程能力，不是单个指标。
2. 可观测性要提前设计，不是出了问题再补。
3. 真正值钱的是减少一类问题，而不是只修一个点。
4. 每次排障都应该为下一次省时间。

## 小任务

选你现在最熟的一条链路，回答：

1. 现在有哪些信号已经能看到？
2. 还缺哪 3 个最关键的观测点？
3. 如果明天线上掉帧，你第一时间最想看什么？

如果你愿意再多做一步，可以继续补：

4. 哪个观测点最应该长期保留下来，作为以后每次回归都要看的一项？

## 复盘题

1. 稳定性为什么不只是“不 crash”？
2. 什么叫可观测性？
3. 为什么“修一个问题”和“减少一类问题”价值不同？
4. 健康检查最应该回答什么？
5. 你现在最想补的一项观测能力是什么？
