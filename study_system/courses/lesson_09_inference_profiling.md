# 第 12 课：已部署推理的 Profiling

## 环境说明

这课必须有可运行的推理链路才适合做真实 profiling。
如果没有环境，先用 latency budget 模板和假想阶段练拆分方法，不把假想数据当成真实性能结论。

## 这节课为什么重要

很多部署问题，最终都要回到一个问题：

时间到底花在哪里？

你可以有很多猜测：

- 模型太大
- 预处理太慢
- 数据拷贝太多
- runtime 不够好

但如果没有 Profiling，这些都只是猜。

这节课的重点，就是把“感觉慢”变成“知道慢在哪里”。

## 学完以后你应该能做到

1. 写出一条部署路径的 latency budget
2. 区分 compute-bound、memory-bound、transfer-bound
3. 理解为什么 steady-state 和初始化成本要分开看
4. 关注 tail latency，而不是只看平均值
5. 建立一个合理的 Profiling 顺序

## 什么叫 Profiling

你可以先把 Profiling 理解成：

“把系统的时间和资源消耗拆开看。”

它不是单纯看一个总耗时，而是要进一步回答：

- 每一段花了多少
- 最慢的是谁
- 最不稳定的是谁
- 哪一段受输入规模影响最大

如果没有这个拆解，优化就很容易走错方向。

## 一个部署路径通常包含哪些时间段

你不要把“推理延迟”只理解成模型执行时间。

一条完整路径通常至少有：

1. 输入获取
2. 预处理
3. 数据搬运到目标执行设备
4. 推理执行
5. 输出搬回
6. 后处理
7. 下游交付

只要你把这些都列出来，你对系统的理解就已经比很多“只看模型时间”的做法更完整了。

## 什么是 Latency Budget

Latency budget 可以先理解成：

“一条链路里每一段大概分掉了多少时间。”

例如：

- 输入 3 ms
- 预处理 8 ms
- 传输 4 ms
- 推理 12 ms
- 后处理 5 ms

这种表比一句“总共 32 ms”有用得多。

因为它能告诉你：

- 哪一段最大
- 哪一段最值得优化
- 哪一段最容易受输入变化影响

## 为什么总耗时不够看

假设两个系统总延迟都 30 ms。

### 系统 A

- 预处理 5
- 推理 20
- 后处理 5

### 系统 B

- 预处理 15
- 推理 10
- 后处理 5

它们总时延一样，但优化方向完全不同。

所以没有 stage breakdown，就谈不上有效优化。

## 三种很常见的瓶颈类型

### Compute-Bound

真正慢在计算执行本身。

你可以粗略理解成：

- 算得太重
- 推理核占主导

### Memory-Bound

慢在内存访问、buffer 组织或数据搬运上。

常见表现：

- 预处理特别重
- resize / color convert 很显著
- cache / 带宽限制明显

### Transfer-Bound

慢在设备之间的数据搬运。

例如：

- CPU 到 GPU
- CPU 到 NPU
- 设备内外来回搬

这类问题特别容易被低估，因为很多人只盯模型执行阶段。

## 为什么 Tail Latency 在部署里一样重要

如果你的平均推理时间很好，但偶发几次很慢，那么：

- 整条链路的实时性会变差
- 相机输入可能积压
- 旧帧可能越来越老

所以 Profiling 时不要只记平均值。
至少要建立这样的习惯：

- 平均值
- 最坏值
- 波动情况

这才更接近真实系统。

## 初始化成本和稳定运行成本要分开

很多部署系统第一次跑会慢一些。
原因可能包括：

- 模型加载
- 内存分配
- 图优化或 engine 初始化

这些开销不应该和 steady-state 的每帧延迟混在一起。

所以你以后做 Profiling 时，一定要区分：

- 冷启动
- 热运行

否则你很容易把问题判断错。

## 一个很实用的 Profiling 顺序

当你面对一条部署链路时，建议先按这个顺序来：

1. 先看端到端总延迟
2. 再拆 preprocess / inference / postprocess
3. 再看传输和 copy
4. 再看 runtime 内部热点
5. 最后再做更细的局部优化

为什么这个顺序重要？

因为很多人会一上来就钻进 runtime 内部细节，但真正的大头可能还在外面。

## 相机场景里怎么理解这一课

如果你把相机输入接到推理链路里，最常见的误判是：

- 看到模型慢，就怪模型

但你更应该问：

1. 输入进来之前是不是已经被处理了很多次
2. 有没有不必要的格式转换
3. 有没有 CPU 和加速器之间来回搬数据
4. 后处理是不是已经比推理还重

这些问题，才符合你“系统工程背景”的优势。

## 一张你以后可以一直复用的 Profiling 表

你可以给每条部署路径做一张表：

| 阶段 | 平均时间 | 最坏时间 | 是否稳定 | 最怀疑的问题 |
| --- | --- | --- | --- | --- |

阶段至少包括：

- 输入
- 预处理
- 传输
- 推理
- 后处理
- 输出

这张表会非常有价值。

## 常见误区

### 误区 1

总延迟够了，就不需要拆阶段。

### 误区 2

推理本体一定是最大瓶颈。

### 误区 3

平均值够好，系统就够好。

### 误区 4

第一次运行很慢，就说明模型一直都慢。

## 你现在先记住这 5 句话

1. Profiling 的目的，是把猜测变成证据。
2. 端到端时间不等于推理本体时间。
3. Latency budget 是定位优化方向的基础。
4. Tail latency 很重要，不能只看平均值。
5. 优化顺序要从整体到局部。

## 小任务

选一条你熟悉的“相机输入 -> 推理输出”路径，先假想写一张 latency budget：

- 输入
- 预处理
- 推理
- 后处理
- 输出

不用完全准确，重点是学会拆。

## 补充干货：Latency Budget 的最小写法

你可以先把一次推理拆成下面这样：

| 阶段 | 时间 |
| --- | --- |
| 数据准备 | x ms |
| preprocess | x ms |
| H2D / D2H 传输 | x ms |
| runtime 执行 | x ms |
| postprocess | x ms |
| 等待与排队 | x ms |

## 补充干货：优化优先级的简单规则

先做下面这三件事，通常不会错：

1. 优先砍掉最大块时间。
2. 优先解决关键路径上的等待。
3. 优先处理会拉高 tail latency 的环节。

## 补充干货：一页速记卡

- Profiling 的目标不是拿到更多数字，而是找到最值钱的优化方向。
- 总耗时只是入口，拆分阶段才有行动意义。
- 计算、传输、等待三类成本必须分开看。
- Tail latency 决定系统“偶尔有多差”，不能只盯平均值。

## 深度学习补充：Profiling 要变成预算表

这课的目标不是会跑 profiler，而是能把推理链路拆成预算。

建议按这条路线学：

1. 先测端到端：从帧到结果到底多久。
2. 再拆阶段：preprocess、copy/transfer、runtime、postprocess、排队等待。
3. 再看分布：平均、P95、P99、最大值分别是多少。
4. 最后决定优化顺序：先优化最大块，还是先处理最差尾部。

常见判断：

1. runtime 很快但端到端慢，优先看数据搬运和预处理。
2. 平均正常但 P99 很差，优先看排队、同步、资源争用。
3. GPU 利用率低，不一定是 GPU 慢，可能是 CPU 喂不饱。
4. batch 提高吞吐，但可能伤害单帧延迟。

掌握标准：

1. 能写出一个 latency budget 表。
2. 能区分 compute time、transfer time、wait time。
3. 能给出一个有证据的优化优先级，而不是凭感觉改代码。
