# 第 6 课：调度、CPU 亲和性、实时性与 Tail Latency

## 为什么这节课很关键

有些系统不是“跑不动”，而是“跑得不稳”。

比如：

- 平均延迟看起来不错
- 但偶发会突然抖一下
- 帧率整体够，但用户还是感觉不顺
- CPU 占用不夸张，但掉帧仍然发生

这时问题常常不在算法，而在调度和 tail latency。

## 这节课的目标

你要学会：

1. 理解 Linux 调度到底在干什么
2. 理解为什么线程和 CPU 之间不是固定绑定关系
3. 理解 CPU affinity 为什么能影响稳定性
4. 区分平均延迟和 tail latency
5. 建立实时性思维，而不是只盯平均值

## 什么是调度

调度器的工作，本质上是在回答一个问题：

“这一刻到底让哪个线程在 CPU 上运行？”

因为线程通常比 CPU 核心多，所以操作系统必须不断决定：

- 谁先跑
- 谁后跑
- 谁继续跑
- 谁先等一会儿

这件事看起来普通，但它直接影响：

- 响应延迟
- 时延抖动
- 上下文切换
- cache 命中情况

## 为什么调度会影响你的相机链路

相机系统通常有多条线程：

- 采集线程
- 处理线程
- 输出线程
- 记录或日志线程
- 其他后台线程

如果调度不稳定，就可能出现：

- 某个关键线程暂时拿不到 CPU
- 某条链路本来应该尽快处理一帧，但被别的任务打断
- hot data 刚进 cache，又因为线程切换被冲掉

这就是为什么“明明 CPU 还有空，系统还是抖”这种现象会出现。

## 什么是 CPU 亲和性 CPU Affinity

CPU affinity 的直觉是：

“尽量让某个线程更稳定地待在某些 CPU 核上运行。”

为什么这会有帮助？

因为线程如果经常在不同核心之间跳来跳去：

- cache 局部性更容易被破坏
- 调度行为更不稳定
- 观察问题也更难

但 affinity 也不是万能的。
如果绑得太死，也可能导致：

- 某几个核过载
- 资源分配不均

所以它是一个“系统调优工具”，不是简单的性能开关。

## 什么是实时性思维

实时性不是“越快越好”。
实时性的重点通常是：

- 在规定时间内稳定完成
- 不要偶发拖太久

换句话说：

- 平均 10 ms
- 偶发 200 ms

这在很多实时场景里比：

- 稳定 20 ms

更糟糕。

因为系统往往怕的不是平均慢，而是偶发很慢。

## 什么是 Tail Latency

Tail latency 可以先粗略理解成：

“最坏那些情况到底有多差。”

比如：

- 大多数帧 20 ms
- 但每 100 帧就有 1 帧 80 ms

这个 80 ms 就是在尾部。

为什么它重要？

- 用户会感觉到卡顿
- 下游模块会读到陈旧数据
- 多传感器对齐会被破坏

所以在系统里，平均值不够，你要看尾部。

## 一个非常常见的误判

很多人看到：

- 平均帧率够
- 平均延迟正常

就认为系统没问题。

但如果：

- p99 很差
- 最大值很夸张
- 抖动很大

那系统在生产环境里通常还是会出问题。

## 关键线程和非关键线程

在真实系统里，不是每个线程都一样重要。

你要学会区分：

- 关键路径线程
- 后台线程
- 日志线程
- 低优先级辅助线程

如果关键路径线程经常被不重要的工作打断，那就是很典型的系统设计问题。

## 为什么上下文切换会让 tail latency 更差

线程切来切去会带来两层成本：

1. 保存和恢复执行状态
2. 破坏 cache 局部性

前者是直接成本。
后者更隐蔽，但在高频数据处理中很常见。

所以你以后看到 tail latency 很差时，要把下面几件事放进怀疑名单：

- 线程过多
- 锁竞争严重
- CPU 亲和性差
- 有不必要的唤醒与睡眠

## 调度问题在相机系统里会怎么表现

常见表现包括：

- 帧处理时间偶尔突然长很多
- 某条流在高负载下抖动明显
- 多路相机时不稳定明显增加
- 系统平均指标还行，但用户体验变差

这些问题经常不是单纯的“算法慢”，而是调度质量不够稳定。

## 一个很实用的工作现场判断法

以后你遇到“偶发抖一下”的问题时，先不要急着说：

- 系统不稳定
- 线程有问题

更好的说法是先分类：

### 情况 A：平均值一直都差

更像长期能力不足。

### 情况 B：平均还行，但偶尔突然很差

更像调度、尾部延迟或争用问题。

### 情况 C：多路同时跑时变差明显

更像关键线程互相影响、绑核不合理、cache 或资源共享问题。

只要先分清这几类，排查就会更有方向。

## 什么时候 CPU affinity 特别值得试

不是所有系统都必须第一时间绑核。
但在下面这些情况下，它很值得尝试：

- 关键线程必须稳定低抖动
- 同一线程反复处理相似数据
- 线程迁移怀疑很重
- 多路并发时偶发抖动明显

你要把 affinity 理解成：

- 一个帮助系统更稳定的调优手段

而不是：

- 一个神奇的性能开关

## 你要形成的判断方式

当你怀疑调度问题时，不要只问：

“这个线程快不快？”

你更应该问：

1. 它是不是关键路径？
2. 它有没有被频繁抢占？
3. 它有没有频繁迁移 CPU？
4. 它有没有和别的线程争同一把锁？
5. 它的最坏情况有多差？

## 常见误区

### 误区 1

平均延迟正常，系统就正常。

### 误区 2

只要 CPU 利用率不高，就不可能有调度问题。

### 误区 3

绑核一定更好。

### 误区 4

实时性等于低平均延迟。

## 你现在先记住什么

先记住这 4 句话：

1. 调度器决定“谁在什么时候用 CPU”。
2. 关键路径线程最怕偶发拿不到 CPU。
3. 实时性关心的是稳定完成，而不是只看平均值。
4. tail latency 往往比平均延迟更接近真实体验。

## 补充干货：Tail Latency 判断表

| 现象 | 更该先怀疑什么 |
| --- | --- |
| 平均值稳定，但偶发长尾很多 | 调度抖动、争用、偶发阻塞 |
| 所有样本都偏慢 | 算力不足、路径太长、常态过载 |
| 某些时段明显变差 | 资源竞争、后台任务、批量操作 |
| 改动线程数后更抖 | oversubscription、切换频繁 |

## 补充干货：给关键线程做的 5 个检查

1. 它是不是经常被非关键线程抢占？
2. 它有没有固定在更稳定的 CPU 上？
3. 它的上游或下游有没有长时间持锁？
4. 它是否承担了太多不属于关键路径的工作？
5. 它出现长尾时，系统同时还在忙什么？

## 补充干货：一页速记卡

- 平均值好看，不代表体验稳定。
- 调度问题常常先体现在尾部，而不是平均值。
- 关键线程要保护，非关键线程要隔离。
- 实时性思维的重点是“最差情况是否仍可接受”。
