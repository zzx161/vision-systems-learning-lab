# 第 1 课：进程、线程、锁与上下文切换

## 先说这节课到底在学什么

这节课不是为了让你背定义。
它真正想解决的问题是：

为什么我们写了多线程程序，系统却不一定更快，反而有时候更慢、更抖、更难查。

对你这种做相机链路、图像处理、系统联调的人来说，这节课特别重要。
因为你的很多问题都不是“算法不行”，而是：

- 线程之间互相等待
- 共享数据设计得不清楚
- 某个锁把整条链路拖慢
- CPU 在不停切换线程

如果这节课你学透了，以后遇到掉帧、延迟抖动、吞吐上不去这种问题，你会比现在更容易看出本质。

## 学完这节课，你应该达到什么程度

这节课学完以后，你不需要变成并发专家。
但你至少应该能做到：

1. 用自己的话说清楚什么是进程，什么是线程
2. 知道线程为什么既有价值又危险
3. 区分 race、contention、deadlock 这三类典型问题
4. 理解上下文切换为什么会影响性能和稳定性
5. 能用“共享状态、等待点、关键路径”这种方式看一条链路

## 先别急着记定义，先建立直觉

### 什么是进程

你可以先把进程想成：

“一个独立运行起来的程序空间。”

比如你打开一个浏览器，打开一个视频播放器，再打开一个终端，这通常就是三个不同的进程。

每个进程通常有：

- 自己的虚拟地址空间
- 自己打开的文件资源
- 自己的一些系统状态

最重要的是：

进程和进程之间默认隔离得比较强。

你可以把它先想成：

- 不同进程像不同房间
- 每个房间有自己的东西
- 别的房间不能直接乱动你的桌子和抽屉

这就是为什么进程隔离更强，也更安全。

### 什么是线程

线程可以先理解成：

“同一个进程内部的多个执行路线。”

也就是说，一个进程里可以有多个线程同时工作。

它们通常共享：

- 同一块虚拟地址空间
- 堆内存
- 全局变量
- 大部分进程级资源

但每个线程仍然有自己的：

- 寄存器状态
- 程序计数器
- 栈

你可以把线程想成：

- 同一个房间里的几个工人
- 他们共用同一块白板、同一张桌子、同一个仓库
- 所以合作很方便
- 但也很容易互相打架

这就是线程最核心的特征：

- 共享带来效率
- 共享也带来风险

## 为什么工程里喜欢用线程

你会在工程里用线程，通常是因为这三种需求：

1. 想并发利用多个 CPU 核
2. 想把不同职责拆开，比如采集、处理、输出
3. 想避免某个慢步骤把整个程序完全卡住

对你做相机链路来说，一个非常典型的模式就是：

- 一个线程收帧
- 一个线程做格式转换或预处理
- 一个线程把结果往下游送

看起来很合理，对吧？

但问题来了：

如果这几条线程之间共享数据太多、锁粒度太粗、等待关系太复杂，那它们就会开始互相拖后腿。

所以重点不是“有没有多线程”，而是“多线程设计得好不好”。

## 线程真正危险在哪里

很多人刚学线程时，会把关注点放在：

- 怎么创建线程
- 怎么 `join`
- 怎么用 mutex

这些当然重要，但真正让工程出问题的，不是 API 本身，而是下面几个结构性问题：

1. 哪些数据是共享的
2. 谁在什么时候写这些共享数据
3. 谁在等谁
4. 哪一段工作在关键路径上

如果这四件事你没想清楚，线程越多，系统越容易出问题。

## 先讲第一个典型问题：Race Condition

Race condition 你可以先这样理解：

多个线程同时读写共享数据，但缺少正确协调，于是结果取决于“谁先谁后”这种偶然时序。

为什么叫 race？

因为线程像在抢时间窗口。

一个简单例子：

- 线程 A 正在更新一帧图像的 metadata
- 线程 B 正在读取这份 metadata 去做后续处理
- 如果 B 在 A 还没写完的时候读到了中间状态
- 那么 B 看到的数据就可能不一致

这类问题最麻烦的地方在于：

- 代码看起来都对
- 问题却不稳定
- 有时能复现，有时不能
- 加一行 log 可能现象都变了

这就是并发 bug 让人头疼的地方。

## 第二个典型问题：Lock Contention

为了解决 race，很多人第一反应是：

“那我加锁。”

这本身没错。
问题是，如果加锁方式不合理，就会引出第二类问题：锁竞争。

锁竞争的直觉是：

多个线程都想进同一扇门，但门一次只允许一个人通过。

于是会发生：

- 某个线程拿到锁
- 别的线程只能等
- 等的线程虽然“活着”，但没有真正做有效工作

在相机和图像链路里，一个特别典型的坑是：

- 所有线程都要更新一个全局队列
- 所有线程都拿同一把大锁

结果就是：

- 线程数量越多
- 等锁的时间越多
- 真正有效工作的比例反而下降

所以你以后看到“线程加了但速度没明显提升”，第一怀疑对象之一就应该是 contention。

## 第三个典型问题：Deadlock

Deadlock 是更严重的一类情况。

你可以把它理解成：

两个或多个线程互相等对方，结果谁都不继续往前走。

最常见的模式是：

- 线程 A 拿着锁 1，等锁 2
- 线程 B 拿着锁 2，等锁 1

于是：

- A 不放
- B 也不放
- 系统卡住

这类问题的本质不是“慢”，而是“没有前进”。

如果你以后遇到：

- 程序没 crash
- CPU 占用不高
- 但系统像卡死一样不动了

那 deadlock 永远值得怀疑。

## 什么是上下文切换 Context Switch

这也是本课最关键的概念之一。

CPU 在任意一个瞬间，只能在某个核心上运行一个线程。
如果有很多线程都处于 runnable 状态，操作系统就要不断切换：

- 先让 A 跑一会儿
- 再让 B 跑一会儿
- 再切回 A

这个切换动作，就叫上下文切换。

为什么叫“上下文”？

因为线程运行时并不只是“正在执行某行代码”，它还有一整套执行状态，例如：

- 当前指令位置
- 寄存器内容
- 栈状态

切换线程时，这些状态要保存和恢复。

## 为什么上下文切换会影响性能

上下文切换的影响有两层。

### 第一层：直接开销

保存状态和恢复状态本身就要花时间。

### 第二层：间接开销

更重要的是，线程切换常常会破坏局部性。

什么意思？

比如：

- 线程 A 刚刚把一批热点数据弄进 cache
- 还没来得及继续处理
- CPU 切去跑线程 B 了
- 等 A 再回来时，原来那批热点数据可能已经不在了

这会让系统：

- 吞吐变差
- 延迟变差
- 抖动更明显

所以你以后看到“CPU 利用率不算高，但时延很不稳”，上下文切换是必须考虑的方向。

## 为什么多线程有时反而更慢

这是你特别值得真正吃透的一件事。

很多人有一个很自然的想法：

- 单线程只能做一件事
- 多线程能并行
- 所以多线程应该更快

但现实中，多线程变慢非常常见。

原因通常包括：

1. 线程之间共享数据太多
2. 锁竞争严重
3. 上下文切换太多
4. 数据复制和同步成本变高
5. 真正瓶颈不在 CPU 算力，而在内存或队列

这也是为什么“线程更多”不等于“系统更强”。

## 用相机场景把它连起来

假设你有这样一条链路：

1. 采集线程收图像
2. 处理线程做格式转换
3. 输出线程发送给下游

如果它们都共享一个全局队列，并且这个队列上有一把大锁，那么可能出现：

- 采集线程要等锁
- 处理线程要等数据
- 输出线程偶发慢了以后，前面所有线程都开始堆积

结果你会看到：

- CPU 不是完全打满
- 但帧延迟还是上去了
- 甚至开始掉帧

这个例子很典型地说明：

系统行为不只取决于“每段代码快不快”，还取决于“这些段之间怎么协调”。

## 这节课最该建立的思维方式

以后你看一个多线程系统，不要只问：

“有几个线程？”

更要问：

1. 哪些数据是共享的？
2. 谁拥有这块数据？
3. 谁在等谁？
4. 哪一段在关键路径上？
5. 哪个地方最可能产生 contention？

这套问法，就是从“写功能”转向“看系统”。

## 常见误区

### 误区 1：线程越多越好

不是。
线程一多，共享、切换、争用都会跟着上来。

### 误区 2：加锁就等于问题解决

不是。
锁能解决正确性的一部分，但可能带来性能问题。

### 误区 3：CPU 不高就说明线程没问题

不是。
系统可能把时间花在等待、切换、争锁上。

### 误区 4：偶发问题不是并发问题

恰恰相反。
并发问题最典型的特征就是偶发、难复现。

## 你现在先记住这 5 句话

1. 进程像不同房间，线程像同一个房间里的多个工人。
2. 线程共享资源，所以沟通方便，但也更容易冲突。
3. Race 是“没协调好”，contention 是“大家都在等同一个东西”，deadlock 是“谁也走不了”。
4. 上下文切换不仅有直接成本，还会破坏 cache 局部性。
5. 多线程系统最重要的问题不是“怎么建线程”，而是“怎么处理共享、等待和关键路径”。

## 这节课的小实验

你不需要先会很多工具。
先做一个很简单的思考实验：

找你现在最熟的一条链路，写下：

1. 有哪些线程
2. 它们共享哪些数据
3. 它们最可能在哪儿等待
4. 哪一把锁如果存在，会最危险

最后写一句话：

“如果这条链路开始抖，我最先怀疑 ______。”

## 补充干货：把线程关系先画出来

很多并发问题不是“代码错得很明显”，而是线程关系没被讲清楚。
你可以先用这种最朴素的方式画一条链：

```text
输入线程 -> 队列 A -> 处理线程 -> 队列 B -> 输出线程
   |           |            |            |
  产出数据     等待空间      竞争 CPU/锁   等待下游
```

只要把这张图画出来，你至少会被迫回答 4 个问题：

1. 哪些线程在生产数据，哪些在线消费数据？
2. 哪些位置可能堆积？
3. 哪些状态被共享？
4. 哪一段在关键路径上？

## 补充干货：多线程排障的 6 个入口

| 入口 | 你在确认什么 | 常见信号 |
| --- | --- | --- |
| 线程数 | 有没有线程膨胀 | 线程数远大于 CPU 核数 |
| CPU 利用率 | 是在忙还是在等 | CPU 不高但延迟很大 |
| 锁等待 | 有没有争用热点 | 某把锁占掉大量时间 |
| 队列长度 | 有没有积压 | 上游快、下游跟不上 |
| 切换频率 | 有没有过度调度 | context switch 偏高 |
| 尾延迟 | 是否偶发很慢 | 平均值正常但卡顿明显 |

## 补充干货：mutex、atomic 和每线程所有权

第 1 课实验现在比较了三种典型写法：

| 写法 | 正确性 | 性能直觉 | 工程含义 |
| --- | --- | --- | --- |
| 共享变量 + mutex | 正确 | 可能很慢 | 所有线程排队进临界区 |
| 共享 atomic 变量 | 正确 | 通常比 mutex 轻，但仍可能慢 | 避免锁阻塞，但共享热点还在 |
| 每线程 local 变量，最后汇总 | 正确 | 通常最好 | 大部分时间没有共享和等待 |

这里最重要的不是背 API，而是建立一个判断：

```text
同步工具能解决正确性，但不一定解决共享热点。
```

`std::atomic` 很适合表达简单的原子状态，例如计数、标志位、轻量统计。
但如果所有线程每一步都写同一个 atomic 变量，这个变量仍然是一个热点。
在多核机器上，多个 CPU 核还要围绕同一份缓存行同步状态，成本可能非常明显。

所以工程上经常会优先考虑：

1. 能不能让每个线程拥有自己的数据？
2. 能不能批量汇总，而不是每次都同步？
3. 能不能把共享状态从关键路径上移走？

## 补充干货：单核、多核和 I/O 等待

这节课还有一个很容易混淆的点：

```text
单核同一瞬间只能执行一个线程。
```

所以如果任务是纯 CPU 计算，单核多线程并不会真正并行计算。
多个线程只是轮流被调度，通常还会多出上下文切换成本。

但 I/O 等待型任务不一样。
当一个线程在等待磁盘、网络、相机帧或 `sleep` 时，它并没有持续占用 CPU。
这时其他线程可以继续运行或进入自己的等待状态，于是等待时间可以重叠。

第 1 课实验里的 sleep-wait 示例就是为了说明这个区别：

```text
CPU 计算型任务：多线程快不快主要看 CPU 核数、共享成本和调度成本。
I/O 等待型任务：即使单核，多线程也可能缩短总墙钟时间，因为等待可以重叠。
```

放到相机系统里，典型例子是：

1. 采集线程等待下一帧到达。
2. 处理线程处理上一帧。
3. 输出线程等待下游发送或写盘。

如果职责切得清楚，这些等待和计算可以自然重叠。
如果共享队列和锁设计得不好，重叠收益又会被等待和竞争吃掉。

## 补充干货：一页速记卡

- 进程强调隔离，线程强调共享。
- 线程带来吞吐潜力，也带来共享状态和等待关系。
- Race 是协调失败，contention 是资源争抢，deadlock 是等待环。
- 上下文切换不只是切线程，还会破坏 cache 局部性。
- 多线程优化的核心不是“线程开出来”，而是“关键路径更短、共享更少、等待更少”。
- `atomic` 比 mutex 轻，但共享热点仍然可能拖慢多核程序。
- 单核不能并行执行 CPU 计算，但可以让多个 I/O 等待重叠。
