# 第 8 课：延迟、抖动、掉帧与同步

## 这节课要解决什么问题

很多系统不是“不能用”，而是“用起来不稳”。

你可能会见到这样的现象：

- 平均帧率看起来可以
- CPU 也没有完全打满
- 功能上也没错

但系统就是会：

- 偶发卡一下
- 有时掉帧
- 输出不平滑
- 多路数据偶发对不齐

这节课要解决的，就是这些“平均值看着还行，但体验和稳定性不行”的问题。

## 学完以后你应该能做到

1. 区分 latency、throughput、jitter、tail latency
2. 知道为什么平均值不够看
3. 能解释常见掉帧的几种路径
4. 知道同步问题并不只有一种
5. 写出一个用于时延问题的诊断模板

## 先把四个最容易混的词分开

### Latency 延迟

延迟指的是：

“一帧数据从起点到终点花了多久。”

比如：

- 从采集到下游收到，花了 28 ms

这就是 latency。

### Throughput 吞吐

吞吐更关心：

“单位时间内能处理多少帧。”

例如：

- 一秒能稳定处理 30 帧

这是吞吐，不是延迟。

### Jitter 抖动

抖动关心的是：

“一帧和下一帧之间的时间稳定不稳定。”

如果：

- 这一帧 20 ms
- 下一帧 21 ms
- 再下一帧 55 ms

那就是明显的抖动。

### Tail Latency 尾部延迟

tail latency 关心的是：

“最差的那部分情况到底有多差。”

这在工程里非常重要，因为用户和下游往往最容易被“少数很差的帧”伤到。

## 为什么平均值经常会骗人

假设有两条链路：

### 链路 A

- 平均 30 ms
- 大部分都在 28 到 32 ms

### 链路 B

- 平均也是 30 ms
- 但有些帧 15 ms，有些帧 70 ms

如果你只看平均值，这两条链路是一样的。
但真实体验完全不一样。

链路 B 更容易：

- 卡顿
- 掉帧
- 让下游处理不稳定

这就是为什么系统工程里，平均值只是起点，不是结论。

## 掉帧为什么会发生

掉帧并不只是“算不过来”。
它至少有几类常见原因：

### 原因 1：处理能力长期跟不上输入

上游一直比下游快，最终总会积压和丢弃。

### 原因 2：短时抖动太大

平均能跟上，但偶发某几帧特别慢，导致队列溢出或时序失效。

### 原因 3：同步策略主动丢帧

有些系统为了保证对齐和时效性，会主动丢弃“来晚了”的帧。

### 原因 4：Buffer / Queue 管理不当

free buffer 不够、回收不及时、队列深度设置不合理，都可能变成掉帧源头。

所以掉帧不等于“算法太慢”，它是一个系统现象。

## 什么是同步

同步这个词特别容易说得很模糊。

你以后要学会区分：

### 线程同步

例如：

- 谁先写
- 谁后读
- 谁等条件满足

### 数据同步

例如：

- 一帧数据和它的 metadata 是否对应

### 多路时间同步

例如：

- 两路相机是否对齐
- 相机和 IMU 是否在同一时间基准上

### 输出时效同步

例如：

- 虽然数据到了，但已经太旧，是否还值得用

这些都叫同步，但本质不是同一类问题。

## 为什么系统“没挂”但体验还是差

因为系统并不只怕 crash。
更常见的情况是：

- 系统还在输出
- 但输出已经不稳定
- 帧还在来
- 但有些帧变得太老
- 平均速率够
- 但 tail 太差

这类问题最难受，因为它不像 crash 那样明显，却会直接影响使用效果。

## Queue 深度为什么既能帮忙，也能害人

queue 深一点，确实可以：

- 吸收一点短时波动
- 减少立刻掉帧

但它同时也可能：

- 增加帧在系统里的停留时间
- 让旧数据越来越老
- 掩盖真实瓶颈

所以以后你看到“把 queue 调大后问题好像少了”，你要追问：

- 是问题解决了
- 还是只是延迟被藏起来了

## 为什么 jitter 往往比平均延迟更伤人

假设一个系统稳定 35 ms，和一个系统在 15 到 60 ms 之间来回波动。

很多时候，后者平均值可能还更好。
但实际体验通常更差。

原因是：

- 用户感知更容易受突刺影响
- 下游算法更难处理不稳定输入
- 时序关系更容易被破坏

这也是为什么工程里必须关注 jitter。

## 用相机场景把这节课落地

假设你的链路是：

- 相机采集
- 预处理
- 下游推理

你看到：

- 平均延迟没明显增加
- 但输出偶发卡一下

那你应该怀疑：

1. 某一段偶发变慢
2. queue 深度在短时增大
3. 某些帧进入下游时已经太旧
4. 某个同步条件在尾部场景下经常失败

这比只说一句“系统抖了”要强很多。

## 一份你以后可以反复用的时延诊断模板

当你怀疑一条链路有时延问题时，先问这 6 个问题：

1. 问题是平均慢，还是偶发慢？
2. 是吞吐不够，还是稳定性不够？
3. 队列有没有明显积压？
4. 帧是“算不过来被丢”，还是“太旧被丢”？
5. 哪一段最容易出现尾部异常？
6. 同步失败到底是线程问题、时间戳问题，还是策略问题？

这 6 个问题能把很多模糊表述变得很具体。

## 常见误区

### 误区 1

平均帧率正常，系统就正常。

### 误区 2

掉帧一定是算力不够。

### 误区 3

同步问题就是“时间没对上”。

### 误区 4

把 queue 调大就是稳定性优化。

## 你现在先记住这 5 句话

1. latency、throughput、jitter、tail latency 不是一回事。
2. 平均值正常，不代表体验正常。
3. 掉帧不一定是算不过来，也可能是策略主动丢弃。
4. queue 可以缓冲波动，也可能藏住问题。
5. 同步问题必须具体到“到底是哪一类同步”。

## 小任务

回想你见过的一次掉帧或卡顿问题，试着回答：

1. 它更像平均慢，还是偶发慢？
2. 它更像 jitter 问题，还是吞吐问题？
3. 如果要加 3 个时间戳，你最想加在哪？

## 复盘题

1. latency 和 throughput 的核心区别是什么？
2. 为什么平均值不够？
3. jitter 为什么会伤害真实体验？
4. 掉帧的常见原因有哪些？
5. 同步问题为什么必须分类型讨论？
