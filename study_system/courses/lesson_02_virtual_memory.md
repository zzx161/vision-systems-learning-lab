# 第 2 课：虚拟内存、Page、Page Fault 与 `mmap`

## 这节课到底在学什么

很多工程师一听“虚拟内存”，第一反应是：

- 很底层
- 很抽象
- 好像和日常写代码没太大关系

但对你这种做相机、图像数据、系统链路的人来说，它其实非常实用。

因为你每天打交道的很多东西，本质上都和内存有关：

- 大 buffer
- 图像拷贝
- 映射内存
- 数据落在哪
- 为什么某次访问突然变慢

这节课真正想解决的是：

为什么“内存”不是一块简单的黑盒，而是一整套影响性能和稳定性的机制。

## 学完这节课，你应该能做到什么

1. 用通俗的话解释什么是虚拟内存
2. 知道 page 和 page fault 是什么
3. 区分栈、堆、匿名内存、映射内存的大致角色
4. 理解为什么大块数据拷贝很贵
5. 对 `mmap` 建立一个够用的工程直觉

## 先建立第一层直觉：程序看到的内存，不等于物理内存本身

程序在代码里写：

- 申请一块数组
- new 一个对象
- 访问一个地址

看起来好像在直接操作“真实内存”。

但实际上，程序通常先看到的是虚拟地址空间。

你可以先把虚拟内存想成：

“操作系统给每个进程提供的一套看起来连续、整齐、私有的地址视图。”

这个视图的好处是：

- 进程之间彼此隔离
- 每个进程不用直接关心物理内存长什么样
- 系统更容易管理和保护内存

## 为什么要搞虚拟内存

如果没有虚拟内存，很多事情会很麻烦：

1. 不同进程之间更难隔离
2. 地址分配会更混乱
3. 系统很难做内存保护
4. 文件映射、共享内存这些机制也更难实现

所以虚拟内存的价值，不只是“高级”，而是让系统：

- 更安全
- 更好管理
- 更容易扩展

## 什么是 Page

Page 可以先理解成：

“内存管理里的基本块。”

也就是说，系统并不是一字节一字节地管理内存，而是按 page 这样的粒度来组织和处理。

你现在不用先记具体大小。
更重要的是记住：

- 内存访问和映射行为，常常是按 page 这个层级发生的

以后你看到：

- page fault
- mapped page
- dirty page

都不要把它当成神秘名词，它们都和“这一块内存页现在处于什么状态”有关。

## 什么是 Page Fault

Page fault 这个词很容易吓人，但你先不要把它理解成“出错”。

更准确地说，它通常表示：

“程序访问了一块当前还没准备好的虚拟页，于是操作系统需要介入处理。”

这件事可能发生在：

- 首次访问某块新分配内存
- 访问映射文件对应的数据
- 某些页还没真正装入可访问状态

所以 page fault 不一定是坏事。
很多时候，它是系统正常工作的组成部分。

真正值得关心的是：

- fault 多不多
- fault 发生在什么路径上
- 它会不会让关键路径突然变慢

## 为什么 page fault 会影响性能

因为一旦 fault 发生，CPU 不能像普通读内存那样直接往下走。
操作系统需要介入，完成一些准备工作。

这意味着：

- 有额外开销
- 关键路径可能被打断
- 某次访问可能突然比平时慢很多

如果你处理的是：

- 高频图像流
- 大块 buffer
- 时延敏感链路

那 page fault 就不是一个纯理论概念，而是可能真实影响你的系统表现。

## 栈和堆到底怎么理解

### 栈 Stack

你可以把栈理解成：

“函数调用过程中自动管理的一块局部工作区。”

特点通常是：

- 生命周期比较短
- 自动管理
- 适合局部变量

### 堆 Heap

堆可以理解成：

“程序主动申请、主动释放的大块更灵活的内存区域。”

特点通常是：

- 生命周期更灵活
- 容量通常更大
- 容易产生复杂的分配与释放行为

对你来说，最需要知道的不是教科书定义，而是：

- 大图像 buffer 更常见地和堆或映射内存相关
- 栈更适合小而短命的局部数据

## 什么是匿名内存

匿名内存可以先粗略理解成：

“不直接绑定某个文件的内存区域。”

例如很多普通的动态内存申请，本质上都更接近匿名内存。

你现在不用特别深挖。
先记住：

- 它不是从某个具体文件直接映射来的
- 它更多是程序自己使用的工作内存

## 什么是 `mmap`

`mmap` 是一类非常有工程味道的机制。

它做的事情可以简单理解成：

“把某个文件或内存对象映射到进程的虚拟地址空间里。”

于是程序就可以像访问普通内存一样去访问它。

为什么这很有用？

因为有些场景里，它能减少传统 `read` / `write` 那种显式拷贝和中间步骤。

但你要记住：

- `mmap` 不是绝对更快
- 它也不是“免费优化”
- 它只是在某些场景下，让数据组织方式更合理

## 为什么图像和视频场景特别怕拷贝

因为数据很大，而且频率很高。

例如一帧图像：

- 分辨率高
- 通道多
- 帧率高

那你每复制一次，就意味着：

- 要消耗内存带宽
- 要占用 CPU 时间
- 要污染 cache

如果每一帧都经历：

1. 驱动到应用一次
2. 格式转换再一次
3. 下游模块再一次

那总成本就会非常可观。

所以你以后面对图像链路时，必须养成一个习惯：

“先怀疑是不是拷贝太多。”

## 这节课最重要的工程视角

很多人看内存问题，只盯“容量够不够”。

但对你来说，更重要的是这三个问题：

1. 数据放在哪
2. 数据怎么被访问
3. 数据被移动了几次

也就是说，真正重要的不是“有没有内存”，而是：

- 地址空间怎么组织
- 页行为如何影响访问
- 数据移动是否成为瓶颈

## 相机场景怎么理解这一课

假设你有一条链路：

- 相机采集
- 用户态接收图像
- 做格式转换
- 送给下游推理

你应该开始学会问：

- 这块图像 buffer 是栈上、堆上，还是映射来的？
- 这块数据有没有被重复复制？
- 第一次访问时会不会触发额外的页准备成本？
- 有没有方式减少中间 buffer？

这些问题比“会不会写 `memcpy`”更有价值。

## 常见误区

### 误区 1：虚拟内存只是为了让内存变大

不是。
它更重要的价值是隔离、管理和抽象。

### 误区 2：page fault 就一定是错误

不是。
很多 fault 是正常机制的一部分。

### 误区 3：`mmap` 一定比 `read` 更快

不是。
它只是在某些访问模式下更合适。

### 误区 4：图像处理慢，主要是算得慢

不一定。
很多时候，慢在“搬得太多”。

## 你现在先记住这 5 句话

1. 程序看到的是虚拟地址空间，不是直接看到物理内存。
2. 系统按 page 这种粒度管理很多内存行为。
3. page fault 不等于灾难，但可能带来明显开销。
4. `mmap` 的价值在于一种不同的数据组织与访问方式。
5. 在图像系统里，内存搬运成本非常值得警惕。

## 补充干货：从地址空间看一遍程序

你可以先把一个普通进程想成下面这样：

```text
高地址
|-----------------------------|
| 栈 stack                    |
|-----------------------------|
| 映射区 mmap / 共享库 / 文件 |
|-----------------------------|
| 堆 heap                     |
|-----------------------------|
| 代码段 / 只读数据           |
|-----------------------------|
低地址
```

这张图最大的意义不是背布局，而是帮助你回答：

- 这块内存是谁申请的？
- 它什么时候才真正分配物理页？
- 它是共享的还是私有的？
- 它是在“移动数据”还是“建立映射关系”？

## 补充干货：虚拟地址到物理内存发生了什么

程序里看到的指针，例如 `0x7ffe...`，通常是虚拟地址。
CPU 真正访问内存时，需要把虚拟地址翻译成物理地址。
这个过程由硬件 MMU 和操作系统维护的页表一起完成。

可以先把流程想成这样：

```text
程序访问虚拟地址
        |
        v
CPU/MMU 查地址翻译信息
        |
        v
找到对应物理页 -> 继续访问
        |
        v
找不到或权限不满足 -> 触发 page fault，让操作系统介入
```

这里有两个重要直觉：

1. 程序不是直接拿着物理地址乱跑，隔离和权限来自这套翻译机制。
2. 虚拟地址看起来连续，不代表背后的物理内存也连续。

这对大 buffer 很重要。
你申请了一块连续的数组，程序视角里它是连续地址。
但物理页可能分散在不同位置，系统通过页表把它们组织成一个连续视图。

## 补充干货：Page Fault 的工程视角

从工程角度看，page fault 更像一句提醒：

“你第一次真正碰到了这块页，系统现在要补齐运行所需的东西。”

| 场景 | 发生了什么 | 直觉代价 |
| --- | --- | --- |
| 首次访问匿名页 | 分配物理页并建立映射 | 首次触碰会变慢 |
| 访问文件映射页 | 把文件页带进内存 | 可能触发 IO 等待 |
| 写时复制 COW | 共享页被拆成私有页 | 额外内存和延迟 |

更细一点，你可以把 page fault 分成两种工程感受：

| 类型 | 直觉 | 例子 |
| --- | --- | --- |
| minor fault | 不一定需要慢速磁盘 IO，系统补一下映射或权限 | 首次触碰匿名内存、COW |
| major fault | 可能需要从磁盘或存储加载数据 | 首次访问还没进内存的文件映射页 |

你现在不用急着记统计口径。
更重要的是知道：

```text
page fault 不是只有一种成本。
有些只是系统补账，有些会牵扯到慢很多的 IO。
```

在实时链路里，最怕的不是“总共发生过 page fault”，而是：

1. page fault 是否发生在每帧处理的关键路径上？
2. 它是否只在启动或预热阶段发生？
3. 它是否导致偶发尾延迟？

## 补充干货：栈、堆、mmap 和共享内存怎么选

先用工程视角看几类内存：

| 区域 | 适合放什么 | 不适合什么 | 常见风险 |
| --- | --- | --- | --- |
| 栈 stack | 小的局部变量、短生命周期状态 | 大图像 buffer | 栈空间有限，生命周期太短 |
| 堆 heap | 动态对象、大 buffer、生命周期灵活的数据 | 高频无节制分配释放 | 分配成本、碎片、首次触碰成本 |
| mmap 区域 | 文件映射、共享库、共享内存、设备映射 | 盲目替代所有 read/write | 成本位置变化，访问模式很关键 |
| 共享内存 | 跨进程传大数据 | 没有所有权和同步设计的数据 | race、同步复杂、生命周期复杂 |

对图像系统来说，一个很实用的判断是：

```text
帧数据通常不适合放栈上。
高频帧数据最好复用 buffer，少做临时分配，少做整帧复制。
```

## 补充干货：大 buffer 拷贝到底贵在哪里

`memcpy` 在代码里只是一行，但运行时它在搬真实数据。
如果一帧图像是 8 MB，复制 3 次，就是 24 MB 的额外数据移动。
如果是 60 FPS：

```text
24 MB * 60 = 1440 MB/s
```

这还没算算法处理，只是搬运。

拷贝的成本主要有三类：

1. CPU 要执行搬运指令。
2. 内存带宽被占用。
3. 大 buffer 会冲刷 cache，让其他热点数据更容易失效。

所以你看一条链路时，不要只问“算法复杂度是多少”，还要问：

1. 每帧经过几个 buffer？
2. 哪些步骤只是改引用，哪些步骤真的复制了整帧？
3. 有没有为了接口方便而多复制了一次？

## 补充干货：访问模式为什么会改变运行时间

内存访问不是“只要地址合法就一样快”。
现代 CPU 和操作系统会利用局部性。

连续访问通常更友好：

```text
buffer[0], buffer[1], buffer[2], buffer[3] ...
```

跳着访问可能更差：

```text
buffer[0], buffer[4096], buffer[8192], buffer[12288] ...
```

原因包括：

1. CPU cache 更喜欢连续、可预测的访问。
2. 硬件预取器更容易猜到下一步要访问哪里。
3. 按 page 跳着访问时，可能更频繁触碰不同页和 TLB 条目。

你不需要现在就把 cache 和 TLB 全部学深。
先记住这个工程结论：

```text
同样的数据量，不同访问顺序可能产生完全不同的性能表现。
```

## 补充干货：mmap 不是“更快版 read”

`read` 的直觉是：

```text
内核把文件数据读出来，拷贝到你提供的用户态 buffer。
```

`mmap` 的直觉是：

```text
把文件的一段映射到进程虚拟地址空间，程序像读内存一样访问它。
```

`mmap` 的价值在于减少某些显式拷贝，让访问方式更自然。
但它不是免费优化。

它可能更合适的场景：

1. 大文件随机访问。
2. 多个进程共享同一个文件映射。
3. 想让操作系统按需把页面带进来。
4. 希望用指针式访问替代手写 read 循环。

它未必更合适的场景：

1. 只顺序读一遍的小文件。
2. 访问模式导致大量 page fault。
3. 对尾延迟非常敏感但没有预热页面。
4. 错误处理和生命周期管理没有设计清楚。

所以更准确的说法是：

```text
mmap 改变了数据进入进程地址空间的方式。
它可能减少拷贝，也可能把成本移动到首次访问页面的时候。
```

## 深度学习补充：这课的学习层次

这课第一轮不要陷入所有内核细节。
你先按四层来学：

1. 地址层：程序看到虚拟地址，不直接看到物理地址。
2. 页层：系统按 page 管理映射、权限、加载和缺页。
3. 数据层：大 buffer 的首次访问、访问模式和拷贝次数都会影响性能。
4. 链路层：图像系统里要把一帧数据从设备到下游的移动路径画出来。

每学一个概念，都把它落到一个问题上：

| 概念 | 要回答的工程问题 |
| --- | --- |
| 虚拟内存 | 这块地址属于哪个进程、哪类区域？ |
| page fault | 这次慢是不是因为首次触碰或按需加载？ |
| heap / mmap | 这块数据是分配出来的，还是映射进来的？ |
| memcpy | 这次整帧复制是否真的必要？ |
| stride | 访问顺序是否破坏局部性？ |

掌握标准：

1. 能解释虚拟地址和物理地址的区别。
2. 能解释 page fault 为什么不一定是错误。
3. 能指出一条图像链路里最可疑的拷贝点。
4. 能谨慎评价 `mmap`，而不是简单说它一定更快。

## 相机链路里的内存检查清单

看到一条图像链路时，可以按这个顺序问：

1. 帧最早在哪里产生？设备、驱动、SDK、用户态？
2. 用户态拿到的是复制出来的数据，还是映射/共享出来的数据？
3. 每一步是否创建了新 buffer？
4. 格式转换是否原地完成，还是又生成一份完整帧？
5. 推理、显示、保存是否各自复制了一份？
6. 是否有启动预热，避免首次处理时 page fault 扎进关键路径？
7. 是否复用 buffer，避免每帧分配和释放？

只要你开始这样问，内存问题就不再是抽象概念，而是一条真实的数据路径。

## 补充干货：一页速记卡

- 程序看到的是虚拟地址，不是裸物理地址。
- Page 是内存管理的基本颗粒。
- Page fault 很多时候不是异常，而是访问真正发生时才补成本。
- `mmap` 的价值在于用映射替代一部分显式搬运。
- 内存问题里最贵的常常不是申请，而是第一次访问和反复拷贝。
- 虚拟地址连续，不代表物理内存连续。
- `read` 和 `mmap` 是不同的数据进入方式，不是简单的快慢关系。
- 图像链路要数清楚每帧被复制了几次。
- 访问模式会影响 cache、预取和页行为。
