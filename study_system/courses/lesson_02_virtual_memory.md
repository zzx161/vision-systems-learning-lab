# 第 2 课：虚拟内存、Page、Page Fault 与 `mmap`

## 这节课到底在学什么

很多工程师一听“虚拟内存”，第一反应是：

- 很底层
- 很抽象
- 好像和日常写代码没太大关系

但对你这种做相机、图像数据、系统链路的人来说，它其实非常实用。

因为你每天打交道的很多东西，本质上都和内存有关：

- 大 buffer
- 图像拷贝
- 映射内存
- 数据落在哪
- 为什么某次访问突然变慢

这节课真正想解决的是：

为什么“内存”不是一块简单的黑盒，而是一整套影响性能和稳定性的机制。

## 学完这节课，你应该能做到什么

1. 用通俗的话解释什么是虚拟内存
2. 知道 page 和 page fault 是什么
3. 区分栈、堆、匿名内存、映射内存的大致角色
4. 理解为什么大块数据拷贝很贵
5. 对 `mmap` 建立一个够用的工程直觉

## 先建立第一层直觉：程序看到的内存，不等于物理内存本身

程序在代码里写：

- 申请一块数组
- new 一个对象
- 访问一个地址

看起来好像在直接操作“真实内存”。

但实际上，程序通常先看到的是虚拟地址空间。

你可以先把虚拟内存想成：

“操作系统给每个进程提供的一套看起来连续、整齐、私有的地址视图。”

这个视图的好处是：

- 进程之间彼此隔离
- 每个进程不用直接关心物理内存长什么样
- 系统更容易管理和保护内存

## 为什么要搞虚拟内存

如果没有虚拟内存，很多事情会很麻烦：

1. 不同进程之间更难隔离
2. 地址分配会更混乱
3. 系统很难做内存保护
4. 文件映射、共享内存这些机制也更难实现

所以虚拟内存的价值，不只是“高级”，而是让系统：

- 更安全
- 更好管理
- 更容易扩展

## 什么是 Page

Page 可以先理解成：

“内存管理里的基本块。”

也就是说，系统并不是一字节一字节地管理内存，而是按 page 这样的粒度来组织和处理。

你现在不用先记具体大小。
更重要的是记住：

- 内存访问和映射行为，常常是按 page 这个层级发生的

以后你看到：

- page fault
- mapped page
- dirty page

都不要把它当成神秘名词，它们都和“这一块内存页现在处于什么状态”有关。

## 什么是 Page Fault

Page fault 这个词很容易吓人，但你先不要把它理解成“出错”。

更准确地说，它通常表示：

“程序访问了一块当前还没准备好的虚拟页，于是操作系统需要介入处理。”

这件事可能发生在：

- 首次访问某块新分配内存
- 访问映射文件对应的数据
- 某些页还没真正装入可访问状态

所以 page fault 不一定是坏事。
很多时候，它是系统正常工作的组成部分。

真正值得关心的是：

- fault 多不多
- fault 发生在什么路径上
- 它会不会让关键路径突然变慢

## 为什么 page fault 会影响性能

因为一旦 fault 发生，CPU 不能像普通读内存那样直接往下走。
操作系统需要介入，完成一些准备工作。

这意味着：

- 有额外开销
- 关键路径可能被打断
- 某次访问可能突然比平时慢很多

如果你处理的是：

- 高频图像流
- 大块 buffer
- 时延敏感链路

那 page fault 就不是一个纯理论概念，而是可能真实影响你的系统表现。

## 栈和堆到底怎么理解

### 栈 Stack

你可以把栈理解成：

“函数调用过程中自动管理的一块局部工作区。”

特点通常是：

- 生命周期比较短
- 自动管理
- 适合局部变量

### 堆 Heap

堆可以理解成：

“程序主动申请、主动释放的大块更灵活的内存区域。”

特点通常是：

- 生命周期更灵活
- 容量通常更大
- 容易产生复杂的分配与释放行为

对你来说，最需要知道的不是教科书定义，而是：

- 大图像 buffer 更常见地和堆或映射内存相关
- 栈更适合小而短命的局部数据

## 什么是匿名内存

匿名内存可以先粗略理解成：

“不直接绑定某个文件的内存区域。”

例如很多普通的动态内存申请，本质上都更接近匿名内存。

你现在不用特别深挖。
先记住：

- 它不是从某个具体文件直接映射来的
- 它更多是程序自己使用的工作内存

## 什么是 `mmap`

`mmap` 是一类非常有工程味道的机制。

它做的事情可以简单理解成：

“把某个文件或内存对象映射到进程的虚拟地址空间里。”

于是程序就可以像访问普通内存一样去访问它。

为什么这很有用？

因为有些场景里，它能减少传统 `read` / `write` 那种显式拷贝和中间步骤。

但你要记住：

- `mmap` 不是绝对更快
- 它也不是“免费优化”
- 它只是在某些场景下，让数据组织方式更合理

## 为什么图像和视频场景特别怕拷贝

因为数据很大，而且频率很高。

例如一帧图像：

- 分辨率高
- 通道多
- 帧率高

那你每复制一次，就意味着：

- 要消耗内存带宽
- 要占用 CPU 时间
- 要污染 cache

如果每一帧都经历：

1. 驱动到应用一次
2. 格式转换再一次
3. 下游模块再一次

那总成本就会非常可观。

所以你以后面对图像链路时，必须养成一个习惯：

“先怀疑是不是拷贝太多。”

## 这节课最重要的工程视角

很多人看内存问题，只盯“容量够不够”。

但对你来说，更重要的是这三个问题：

1. 数据放在哪
2. 数据怎么被访问
3. 数据被移动了几次

也就是说，真正重要的不是“有没有内存”，而是：

- 地址空间怎么组织
- 页行为如何影响访问
- 数据移动是否成为瓶颈

## 相机场景怎么理解这一课

假设你有一条链路：

- 相机采集
- 用户态接收图像
- 做格式转换
- 送给下游推理

你应该开始学会问：

- 这块图像 buffer 是栈上、堆上，还是映射来的？
- 这块数据有没有被重复复制？
- 第一次访问时会不会触发额外的页准备成本？
- 有没有方式减少中间 buffer？

这些问题比“会不会写 `memcpy`”更有价值。

## 常见误区

### 误区 1：虚拟内存只是为了让内存变大

不是。
它更重要的价值是隔离、管理和抽象。

### 误区 2：page fault 就一定是错误

不是。
很多 fault 是正常机制的一部分。

### 误区 3：`mmap` 一定比 `read` 更快

不是。
它只是在某些访问模式下更合适。

### 误区 4：图像处理慢，主要是算得慢

不一定。
很多时候，慢在“搬得太多”。

## 你现在先记住这 5 句话

1. 程序看到的是虚拟地址空间，不是直接看到物理内存。
2. 系统按 page 这种粒度管理很多内存行为。
3. page fault 不等于灾难，但可能带来明显开销。
4. `mmap` 的价值在于一种不同的数据组织与访问方式。
5. 在图像系统里，内存搬运成本非常值得警惕。
