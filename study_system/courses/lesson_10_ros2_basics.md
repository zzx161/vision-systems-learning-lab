# 第 13 课：面向视觉工程师的 ROS2 基础

## 环境说明

这课可以先用纸面拓扑学习 ROS2 的系统组织方式。
如果没有 ROS2 环境，不讲具体命令细节；等安装环境后再做 topic、node、QoS 的实操。

## 为什么这节课值得你学

你不一定要立刻去做机器人算法。
但如果你想给自己准备一条更长线的职业路线，ROS2 是非常值得了解的基础。

原因很简单：

- 它是很多机器人系统的常见基础设施
- 它非常适合表达“多模块、多节点、数据流、消息传递”的系统结构
- 它和你现在做相机链路的经验有很强的可迁移性

所以你学 ROS2，不是为了转行去卷新概念，而是为了把你现在的系统经验迁移到另一个赛道。

## 学完以后你应该能做到

1. 知道 node、topic、service、action 分别在干什么
2. 能把一条相机链路映射成 ROS2 风格系统
3. 理解 QoS 为什么对视觉数据很重要
4. 认识 launch、参数、节点拆分这些工程概念
5. 能画出一张简单的 ROS2 系统图

## 先建立第一层直觉：ROS2 更像“系统组织方式”，不只是一个库

很多人刚接触 ROS2，会把它理解成：

- 一个 API
- 一个框架

这都不算错，但不够。

对你更有用的理解是：

ROS2 是一种把复杂机器人系统拆成多个可协作模块的组织方式。

它帮你做的事情包括：

- 定义模块边界
- 定义消息流
- 定义启动方式
- 定义不同模块之间的通信方式

这和你现在做相机系统时会面对的问题，其实很像。

## 什么是 Node

Node 可以先理解成：

“系统中的一个功能单元。”

例如：

- 相机采集节点
- 图像预处理节点
- 推理节点
- 结果发布节点

Node 的价值在于：

- 职责更清楚
- 边界更清楚
- 方便独立调试和替换

## 什么是 Topic

Topic 可以先理解成：

“一条命名的数据流通道。”

比如：

- `/camera/image_raw`
- `/camera/image_rect`
- `/detections`

一个节点可以往 topic 里发数据，另一个节点可以从这个 topic 订阅数据。

所以 topic 的本质是：

- 让数据在模块之间流动

## 什么是 Publisher 和 Subscriber

Publisher：

- 把消息发到某个 topic 上

Subscriber：

- 从某个 topic 接收消息

如果你拿相机系统类比，很像：

- 上游模块把一帧图像“发布”出去
- 下游模块决定是否“订阅”它

这套模型为什么有用？

因为它强迫你清楚表达：

- 数据是什么
- 谁在发
- 谁在收

## 什么是 Service 和 Action

这两个你现在不用学得很深。
先建立一个直觉就够：

### Service

更像：

- 请求一下
- 马上给个回应

适合做配置、控制、查询类事情。

### Action

更像：

- 这是一个需要时间完成的目标
- 中间还会持续反馈进度

在你当前阶段，topic 的理解优先级最高，service / action 先知道大概用途就够了。

## 为什么 QoS 特别重要

QoS 是 ROS2 里一个非常有工程味道的点。

你可以先把它理解成：

“系统如何看待消息传递的可靠性、时效性和缓存策略。”

对视觉链路来说，这很关键。

因为图像数据通常：

- 量大
- 频率高
- 更在意“新鲜”

这时你就要考虑：

- 是不是每一帧都必须保留？
- 还是宁可丢旧帧，也要尽快处理新帧？

这其实和你现在在相机链路里考虑 queue、延迟、掉帧，本质上是连着的。

## Launch 为什么重要

机器人系统通常不只是一个进程。

你可能要一起启动：

- 相机节点
- 预处理节点
- 推理节点
- 控制或输出节点

这时候 launch 的价值就出来了：

- 统一启动
- 统一参数
- 统一连接关系

它背后的本质是：

“把系统作为系统来组织，而不是只盯一个程序。”

## 为什么你做相机系统的人学 ROS2 很有优势

因为你本来就已经在思考这些问题：

- 数据从哪里来
- 数据怎么流
- 哪里可能堵
- 哪个模块在关键路径上

ROS2 只是把这套思考方式显式化、系统化了。

所以你不是从零学一门完全陌生的东西，而是在迁移已有能力。

## 一条很典型的 ROS2 风格视觉链路

你可以先把它想成：

1. 相机节点发布原始图像
2. 预处理节点订阅图像并输出处理结果
3. 推理节点订阅处理后的图像
4. 结果节点发布检测或状态

这条链路看起来简单，但它已经包含了很多关键问题：

- 节点怎么拆
- 消息格式怎么定
- QoS 怎么选
- 哪些步骤可以分开
- 哪些步骤可能要尽量靠近

## 节点拆得越细越好吗

不一定。

拆得细的好处：

- 清晰
- 解耦
- 好替换

拆得太细的问题：

- 通信开销增加
- 数据拷贝增加
- 调试路径变长

所以工程上永远不是“越细越先进”，而是“边界是否划得合适”。

## 相机场景和 ROS2 的连接点

你以后完全可以用 ROS2 的思路重新看你现在的链路：

- 哪些步骤其实天然是独立节点
- 哪些步骤必须尽量靠近，避免拷贝
- 哪些地方应该加时间戳
- 哪些 topic 的 QoS 应该偏“最新优先”

这会帮助你从“单模块开发”过渡到“系统架构思维”。

## 常见误区

### 误区 1

ROS2 只是机器人专用语法糖。

### 误区 2

节点拆得越细越好。

### 误区 3

QoS 只是高级配置项，先不用关心。

### 误区 4

学 ROS2 就等于要去卷机器人算法。

## 你现在先记住这 5 句话

1. ROS2 更像系统组织方式，而不只是一个库。
2. Node 是功能单元，topic 是数据流。
3. QoS 决定消息传递的可靠性和时效性策略。
4. 节点拆分不是越细越好，而是要看代价和边界。
5. 你的相机系统经验，本来就很适合迁移到 ROS2 思维。

## 小任务

把你最熟的一条相机链路，试着改写成 ROS2 风格：

1. 哪些部分可以当 node？
2. 哪些数据流可以当 topic？
3. 哪一条 topic 最在意新鲜度？

## 复盘题

1. Node 和 topic 分别是什么？
2. QoS 为什么对视觉数据特别重要？
3. Launch 在系统里主要解决什么问题？
4. 节点拆分为什么有代价？
5. 你当前工作中哪条链路最适合用 ROS2 思维来表达？

## 下次我会怎么带你学

等你学到这一课时，我可以直接带你把一条相机链路画成 ROS2 系统图，让它从“概念”变成你能看懂、能讲的工程图。

## 补充干货：先学会画 ROS2 拓扑

一条最小 ROS2 视觉链路可以先画成：

```text
camera_node -> preprocess_node -> inference_node -> postprocess_node -> viewer_node
```

你先不用着急写代码。
只要先把节点边界和 topic 画清楚，很多架构问题就会先暴露出来。

## 补充干货：QoS 的快速判断

| 场景 | 更偏向什么策略 |
| --- | --- |
| 强调最新结果 | 更倾向保留最新、允许丢旧消息 |
| 强调不漏消息 | 更倾向可靠传输 |
| 高频传感器流 | 更要小心队列积压 |
| 调试和录包 | 更关注可重放和一致性 |

## 补充干货：一页速记卡

- ROS2 的价值在于把系统边界和数据流讲清楚。
- Node 是职责边界，topic 是数据连接。
- QoS 会直接影响时效性、可靠性和积压风险。
- 拓扑图先画清楚，代码会好写很多。

## 深度学习补充：ROS2 先学系统边界

这课第一轮不要追求写很多节点。
先学会用 ROS2 表达系统边界和数据流。

建议按这条路线学：

1. 先画 node：每个节点负责什么，不负责什么。
2. 再画 topic：数据从哪里来，到哪里去，频率是多少。
3. 再定 message：字段、时间戳、frame id 是否足够。
4. 再看 QoS：最新优先、可靠优先、队列多深。

视觉系统里特别要注意：

1. 高频图像 topic 很容易积压。
2. 可靠传输不总是适合实时视频流。
3. 时间戳和坐标系字段缺失，会让后面同步和标定很痛苦。
4. node 过细会增加通信成本，过粗会降低可观测性和复用性。

掌握标准：

1. 能画出一个视觉 ROS2 pipeline 拓扑。
2. 能解释一个 topic 应该用什么 QoS 倾向。
3. 能说明 node 边界如何影响调试和部署。
