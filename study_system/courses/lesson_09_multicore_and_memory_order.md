# 第 9 课：多核、一致性、内存序与扩展性

## 为什么这节课值得学

很多工程师学多线程时，只看到“可以并发”。
但真正难的是：

- 为什么线程多了以后收益没线性增长
- 为什么共享数据一多，系统突然变差
- 为什么有些 bug 很偶发，像随机的一样

这些问题背后，往往都和多核、一致性、内存序有关。

## 这节课的目标

你要学会：

1. 理解多核系统为什么复杂
2. 理解 cache coherence 为什么必须存在
3. 理解共享数据为什么会限制扩展性
4. 对 memory ordering 建立基本直觉
5. 解释为什么“线程变多但速度没上去”

## 多核系统到底复杂在哪里

单核时，一个时刻只有一个执行点。
很多事情虽然麻烦，但行为比较线性。

多核时，多个核心可以同时读写共享数据。
这时问题就来了：

- 每个核心都有自己的 cache
- 谁先看到哪个值，不一定完全同步
- 共享变量的变化，必须有机制让别的核心最终看到

所以多核带来的不是“免费加速”，而是“更复杂的一致性问题”。

## 什么是一致性 Cache Coherence

你可以先这样理解：

多个核心的 cache 里，不能长期保存互相矛盾的同一份共享数据副本。

否则会出现：

- 核 A 看到旧值
- 核 B 看到新值
- 程序行为变得难以预测

一致性机制的作用，就是尽量保证共享数据不会一直“各看各的版本”。

## 为什么一致性是有成本的

一致性不是白来的。
为了让多个核心对共享数据达成相对合理的视图，系统要付出：

- 通信成本
- 同步成本
- cache line 失效与更新成本

所以共享写入一多，多核扩展性就容易下降。

## 什么是扩展性

扩展性不是“线程越多越快”。
扩展性真正问的是：

“增加资源后，性能到底能增加多少？”

如果：

- 1 个线程 1x
- 2 个线程 1.8x
- 4 个线程 2.2x
- 8 个线程 2.3x

那就说明系统扩展性很差。

背后的常见原因是：

- 锁竞争
- 共享写入
- false sharing
- 内存带宽瓶颈
- 同步等待

## 什么是内存序 Memory Ordering

这是一个容易让人害怕的词。

你不用一开始就背标准定义。
你只需要先建立一个直觉：

“程序里写出来的先后顺序，不一定总是被所有核心按完全相同的顺序观察到。”

为什么？

因为 CPU 和编译器都会为了性能做重排与优化。

在单线程下，这通常没问题。
但在多线程共享数据时，如果你没有正确同步，就可能出事。

## 为什么同步原语重要

mutex、condition variable、atomic 不是“语法工具”，而是“约束可见性和顺序”的工具。

没有这些同步手段时：

- 你以为另一个线程已经看到新值了
- 但它可能还没看到

这就是为什么很多并发 bug 会表现成：

- 偶现
- 不稳定
- 换机器后更明显

## 一个非常重要的现实判断

如果一个系统大量依赖共享可变状态，那么它的扩展性通常很难好。

所以强工程思维不是只想着“怎么加线程”。
而是要问：

- 能不能减少共享状态
- 能不能把数据分片
- 能不能把写共享变成读共享
- 能不能降低同步频率

## 相机场景里怎么理解这件事

例如：

- 多线程同时处理多个 frame
- 但所有线程都要更新同一个全局统计结构
- 或者都要争一个全局 buffer 池

这时线程虽然多了，但共享点仍然存在。
共享点就会限制扩展性。

所以你以后看到“线程加了，但几乎没变快”，要优先怀疑：

- 共享资源
- 锁
- 带宽
- 一致性开销

## 一个特别值得记住的判断顺序

以后你面对“为什么多线程扩不起来”时，先按这个顺序想：

1. 线程是不是都在干独立工作？
2. 它们之间有没有频繁共享写入？
3. 有没有全局锁或全局队列？
4. 是不是在争同一类热点 cache line？
5. 是不是已经碰到带宽上限？

这个顺序很重要，因为很多人会过早怀疑 CPU 不够强，但更常见的真实原因是共享太多。

## 为什么这节课和你未来成长很相关

如果你以后想从“相机接入开发”走向“系统工程师”，多核扩展性是非常关键的一步。

因为系统工程师不只是会写并发代码，还要能回答：

- 这条链路为什么扩不起来
- 共享点到底在哪里
- 怎样的改法才更可扩展

这类能力在复杂系统里非常值钱。

## 常见误区

### 误区 1

多核机器上，多线程一定接近线性加速。

### 误区 2

只要没有死锁，多线程设计就没问题。

### 误区 3

atomic 是“更轻的锁”，用了就一定更快。

### 误区 4

共享状态只是写代码方便，对性能影响不大。

## 你现在先记住什么

先记住这 4 句话：

1. 多核不是免费性能，而是带来共享一致性问题。
2. 一致性有成本，尤其在共享写入多时。
3. 扩展性差，常常不是“核心不够”，而是“共享太多”。
4. 没有正确同步时，多线程程序的行为可能很不稳定。

## 小任务

1. 它的共享状态有哪些？
2. 哪些是读共享，哪些是写共享？
3. 哪个共享点最可能限制扩展性？

如果你愿意更进一步，再写一句：

“如果我要先拆掉一个共享点，我最先想拆 ______。”

## 下次我带你怎么学

等你学到这里时，我可以陪你做一件很有价值的事：

把你某条多线程链路里的共享点逐个画出来，找出真正限制扩展性的地方。

## 补充干货：多核扩展为什么会卡住

一个任务扩不起来，常见不是因为“核心不够多”，而是下面这些东西没有一起跟上：

1. 共享状态太多。
2. 每个核都在抢同一块数据。
3. 同步点太密。
4. 内存带宽先撞墙了。

## 补充干货：看到 atomic 不要自动等于高性能

`atomic 比 mutex 轻，所以一定更快。`

这句话只对了一半。

atomic 省掉了某些锁路径，但如果它落在高频共享热点上，一样可能拖垮扩展性。

## 补充干货：一页速记卡

- 多核不等于线性加速。
- 一致性是为了正确性付出的硬件代价。
- 扩展性问题常常先表现为共享热点，而不是 CPU 不够。
- atomic 和 mutex 解决的是不同层次的问题，不是谁天然更高级。

## 深度学习补充：多核优化先找共享热点

这课不是为了把内存序所有细节一次学完。
第一轮目标是理解：多核加速的敌人常常不是计算量，而是共享状态和一致性成本。

建议按这条路线学：

1. 先回顾第 1 课的 atomic 计数器实验。
2. 再理解 cache coherence：多个核心必须对同一份数据的读写达成一致。
3. 再看扩展性：线程数增加时，吞吐是否线性增加，还是很快撞墙。
4. 最后再碰 memory order：先知道它解决“可见性和顺序”问题，不急着背每个枚举。

工程上优先检查：

1. 有没有所有线程都频繁写的全局计数器？
2. 有没有所有 worker 都抢的队列头尾？
3. 有没有共享状态和高频数据放在同一 cache line？
4. 有没有每帧都做全局同步的 barrier？

掌握标准：

1. 能解释为什么 atomic 不是免费操作。
2. 能识别“CPU 核很多但加速不上去”的共享热点原因。
3. 能提出 per-thread shard、批量汇总、减少同步频率这类改法。
