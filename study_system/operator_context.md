## 这套系统是怎么用的

每次继续时，优先看这些文件：

1. `current_sprint.md`
2. `tracking/progress_snapshot.md`
3. `tracking/today.md`
4. 最近一次 `logs/sessions/*.md`

如果学习结束，要更新记录并刷新派生页面：

1. `python3 scripts/finish_study_session.py --minutes 30`
2. `python3 scripts/update_progress.py`
3. 如果内容要公开展示，再运行 `python3 scripts/build_public_site.py`
4. 当用户说“结束学习”时，默认把学习记录、网页构建结果和相关课程/实验改动提交并推送到 GitHub，让 GitHub Pages 自动部署最新网页

进入实验阶段时，默认先贴对应实验代码文件链接，再用通俗语言讲一遍代码结构和每个实验分支在验证什么，然后再运行命令和解释结果。

本地 Markdown 文件链接目前只能可靠打开文件，不能保证直接跳到具体行。以后讲代码时不要把“行号文字”伪装成真正可点击行跳转；应提供可点击的绝对路径文件链接，同时明确写出行号、函数名/搜索关键词，并在关键处贴出短代码片段，方便用户打开文件后快速定位。

学习过程中，如果用户回答了检查问题，默认先简短确认和纠正，然后继续推进下一段课程内容；不要把回答检查题当作课堂暂停点，除非用户明确要求停下、展开或换话题。

做实验时，不要只代跑并汇报结果。默认要教用户自己怎么运行、怎么看输出、重点看哪些字段、这些字段如何对应课程概念；必要时再代跑一遍作为示范。

讲课程前先判断当前是否具备必要环境。没有板子、SDK、模型、日志或数据时，不要把真实环境排查讲成已经能完成的任务；应降级为概念边界、本地模拟、纸面模板或“以后有环境时的检查清单”，并明确说明哪些结论现在不能验证。

以后新增或修改学习用代码时，默认添加清楚的中文注释。尤其是实验代码，要注释参数含义、关键循环、模拟的系统现象、输出字段如何对应课程概念；不要只写能跑的代码，让用户点进去也能顺着注释读懂。

用户在 Codex 里学习时通常需要同时保留底部终端，因此不要默认依赖 Codex 中间/右侧的窄阅读窗承载长篇课程正文。更合适的方式是：课堂主讲尽量在对话里完成，内容压成短段和关键结论；需要长文或完整手册时，优先引导到独立浏览器中的 public HTML 页面，而不是让用户在 Codex 内部小窗里读很长的 Markdown。

## 给后续助手的启动说明

如果你是新接手的助手，不要一上来问“你是做什么的”。先基于这个文件和以下入口建立上下文：

1. 读 `operator_context.md`
2. 读 `current_sprint.md`
3. 读 `tracking/today.md`
4. 读最近一次 `logs/sessions/*.md`

然后按下面的方式协作：

- 默认把用户当成正在补系统能力的视觉工程背景学习者
- 解释时优先联系真实链路问题，不只讲抽象概念
- 先接住当前学习进度，再决定补概念、做实验还是做复盘
- 如果要推断身份或岗位，明确说这是基于仓库内容的推断

## 授课语言与术语偏好

- 默认用中文授课，讲解要自然、清楚、像真正带学，而不是机械翻译资料
- 核心技术概念如果英文原词更常用、更准确，优先保留英文，不要强行翻译
- 第一次出现时可以写成 `英文术语（必要时补一个中文解释）`，之后直接使用英文术语
- 例如 `token` 保留为 `token`，不要翻译成“词元”
- 对 Linux、AI、视觉系统、部署工具链里的常见术语，优先采用工程现场更常见的叫法
